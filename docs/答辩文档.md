# AgentFuzzer 2.0 - 完整技术答辩文档

## 一、项目概述

### 1.1 项目名称

**AgentFuzzer 2.0** —— 面向 AI Agent 的自动化漏洞扫描、行为审计与容器化安全靶场平台

### 1.2 解决的核心问题

在企业 AI Agent 部署中，攻击者可能通过以下方式操控 Agent：

- **Prompt 注入** — 让 Agent 忽略安全限制
- **数据泄露** — 诱导 Agent 读取并外发敏感信息
- **命令执行** — 让 Agent 执行系统命令
- **权限提升** — 越权访问管理员接口

### 1.3 三大子系统

| 子系统             | 定位    | 核心能力                                   |
| --------------- | ----- | -------------------------------------- |
| **AgentGuard**  | 被动审计  | 单次输入审计、规则检测、风险评分、可信存证报告                |
| **AgentFuzzer** | 主动评测  | 靶标注册、载荷库、批量 Fuzzing、漏洞聚合               |
| **CASS**        | 容器化靶场 | Docker 运行受控 Agent、生成 AgentTrace、五层防线分析 |

### 5.1 行为提取器 —— behavior_extractor.py

**混合提取策略：**

```python
async def extract_behavior_chain(input_type, content):
    # 1. 尝试 LLM Agent 提取
    agent_result = await extract_by_agent(input_type, content)

    # 2. 同时运行 Fallback 提取（关键词匹配）
    fallback_result = extract_by_fallback(input_type, content)

    # 3. 智能合并
    if agent_result 有效:

            return agent_result
```

#### 5.1.2 Fallback 提取器 —— fallback_extractor.py

**关键词映射表：**

| 维度              | 关键词示例                         | 映射值     |
| --------------- | ----------------------------- | ------- |
| **Action**      | 读取、查看、获取 → read               | 40+ 关键词 |
|                 | 发送、上传、外传 → send               |         |
|                 | 删除、清空、格式化 → delete            |         |
|                 | 执行、运行、提权 → execute            |         |
|                 | 忽略、绕过、扮演 → override           |         |
|                 | 输出、泄露、告诉我 → leak              |         |
| **Data Type**   | 密码、token、api_key → credential | 30+ 关键词 |
|                 | 身份证、手机号 → personal_info       |         |
| **Tool**        | shell、bash、cmd → shell        | 35+ 关键词 |
|                 | 数据库、SQL → database            |         |
|                 | 邮件、邮箱 → email                 |         |
| **Permission**  | 同学、他人、未授权 → unauthorized      | 15+ 关键词 |
| **Destination** | 外部、邮箱、远程 → external           | 10+ 关键词 |

**智能推断逻辑：**

1. **分词**：按标点、连接词分割为多步骤
2. **关键词匹配**：每个步骤独立匹配
3. **字段推断**：
   - action 为空 → 从 tool 推断（shell→execute, database→write）
   - tool 为空 → 从 action 推断（execute→shell, send→network）
   - data_type 为空 → 从 action + content 推断
4. **后处理**：检测 shell 命令、SQL 语句、远程脚本等模式

**后处理规则：**

- 检测到 `rm|chmod|sh|bash|curl|wget` → tool=shell
- 检测到 `DELETE|DROP|TRUNCATE` + 表名 → tool=database
- 检测到 `curl ... | sh` → data_type=code, destination=external
- 检测到 敏感数据关键词 → 自动识别 data_type

### 5.2 规则引擎 —— rule_engine.py

**双层规则体系：**

#### 第一层：YAML 规则库（85+ 条）

**规则结构：**

```yaml
- id: R001
  name: 敏感凭证读取
  category: 隐私泄露
  type: node          # node 或 chain
  condition:
    action: ["read"]
    data_type: ["credential", "password", "token", "api_key"]
  score: 35
  level: high
  advice: 禁止 Agent 访问凭证类数据...
```

**规则类型：**

- **Node 规则**：匹配单个行为节点（action + data_type + tool + permission + destination）
- **Chain 规则**：匹配行为链序列（多步骤组合）

**规则分类（12 大类）：**

1. 隐私泄露（ credential/password/token 读取）
2. 数据外传（敏感数据发送到外部）
3. 系统安全（shell 命令执行）
4. 越权访问（未授权操作）
5. 提示词注入（override 行为）
6. 模型安全（系统提示词泄露）
7. 代码执行（远程脚本下载执行）
8. 权限提升（sudo/root 操作）
9. 数据安全（SQL 注入、数据库删除）
10. 社会工程学（诱导信息泄露）
11. 供应链风险（pip/npm 安装未知包）
12. 正常任务排除（负分规则，降低误报）

#### 第二层：内容直接检测规则（20 条）

**设计目的：** 弥补 LLM 提取失败时的检测空白

### 5.3 风险评分引擎 —— risk_engine.py

#### 5.3.1 节点风险计算

**五维度评分模型：**

```python
def calculate_node_risk(node):
    tool_risk = TOOL_RISK_MAP.get(node.tool, 30)      # 权重 18%
    data_risk = DATA_RISK_MAP.get(node.data_type, 30)  # 权重 22%
    action_risk = ACTION_RISK_MAP.get(node.action, 30) # 权重 22%
    dest_risk = DEST_RISK_MAP.get(node.destination, 60) # 权重 15%
    perm_risk = PERMISSION_RISK_MAP.get(node.permission, 50) # 权重 18%
    uncertainty = 20 if permission == "unknown" else 5  # 权重 5%

    return weighted_sum
```

**风险映射表：**

| 维度          | 最高风险值           | 对应场景   |
| ----------- | --------------- | ------ |
| Tool        | shell=85        | 系统命令执行 |
| Data        | password=95     | 密码读取   |
| Action      | delete=90       | 删除操作   |
| Destination | external=90     | 数据外发   |
| Permission  | unauthorized=90 | 未授权访问  |

#### 5.3.2 行为链风险计算

**概率累积模型：**

```python
def calculate_chain_risk(nodes):
    risk_product = 1.0
    for node in nodes:
        nr = calculate_node_risk(node)
        risk_product *= (1 - nr / 100)
    return 100 * (1 - risk_product)
```

**原理：** 假设每个节点的风险是独立事件，链的整体风险 = 1 - 所有节点都安全的概率

#### 5.3.3 组合攻击检测

**预设组合模式：**

| 组合                           | 加分  | 说明       |
| ---------------------------- | --- | -------- |
| password + send_external     | +30 | 密码外发     |
| credential + send_external   | +30 | 凭证泄露     |
| download + execute           | +25 | 下载后执行    |
| override + leak              | +25 | 注入后泄露    |
| unauthorized + personal_info | +20 | 越权访问个人信息 |
| delete + system_file         | +20 | 删除系统文件   |

#### 5.3.4 动态权重调整

```python
if normalized_rule < 0:      # 正常任务（负分规则命中）
    chain_weight = 0.20
    rule_weight = 0.60       # 信任规则判断
elif normalized_rule > 50:   # 高危攻击
    chain_weight = 0.45
    rule_weight = 0.35       # 规则和行为链并重

    chain_weight = 0.55
    rule_weight = 0.25
```

**最终评分公式：**

```
FinalScore = chain_weight × ChainRisk 
           + rule_weight × RuleScore 
           + 0.05 × DefenseBreachRisk 
           + 0.10 × DataFlowRisk 
           + 0.05 × Uncertainty 
           + 0.05 × ComboBonus
```

| L1  | Prompt 防线 | System Prompt 覆盖/泄露/角色劫持 | CWE-77, CWE-200 |
| L2  | 意图防线      | Agent 意图被劫持、合理化危险操作      | CWE-693         |
| L3  | 权限防线      | 越权 API 调用、管理员操作          | CWE-269         |
| L4  | 数据防线      | 敏感数据泄露、数据外发              | CWE-200         |
| L5  | 执行防线      | 危险 Shell 命令、SQL 删除、格式化   | CWE-78, CWE-912 |

**检测方法：**

1. **YAML 规则**：可配置的检测规则（keyword_match, regex_match, similarity_check, sensitive_data_scan, permission_check, llm_analysis）
2. **硬编码逻辑**：针对各层的经典攻击模式

### 5.6 可信存证链 —— evidence_chain.py

**SM3 哈希链：**

```python
async def append_evidence_record(previous_hash, record):
    # 1. 提取关键字段
    payload = {
        "input_type": record.input_type,
        "risk_score": record.risk_score,
        "risk_level": record.risk_level,
        "policy_action": record.policy_decision.action,
        "matched_rule_ids": sorted([r.id for r in matched_rules]),
    }

    # 2. 规范化 JSON（排序键、去空格）
    canonical = json.dumps(payload, sort_keys=True, separators=(",", ":"))

    # 3. 组合哈希
    combined = previous_hash + canonical

    # 4. SM3 哈希（gmssl 库，fallback 到 SHA-256）
    return sm3_hash(combined)
```

**链式验证：**

```python
async def verify_evidence_chain(records):
    prev_hash = ""
    for record in records:
        expected = await append_evidence_record(prev_hash, record)
        actual = record.record_hash
        if expected != actual:
            return False  # 链断裂
        prev_hash = actual
    return True
```

### 5.7 多 Agent 协作系统 —— multi_agent_system.py

**三层协作架构：**

```
┌─────────────────┐
│  Orchestrator   │ ← 总调度，判断任务复杂度
│    Agent        │   simple/medium/complex
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌─────────┐ ┌─────────┐
│  Risk   │ │ Policy  │
│ Analyst │ │ Advisor │
└─────────┘ └─────────┘
```

**1. Orchestrator Agent**

- 输入：用户输入内容
- 输出：JSON 决策（需要哪些 Agent、复杂度等级）
- 判断标准：
  - simple：单步骤、意图明确 → 只需 Extractor
  - medium：多步骤、意图模糊 → Extractor + RiskAnalyst
  - complex：社会工程学、权限提升 → 全部 Agent

**2. Risk Analyst Agent**

- 输入：原始内容 + 行为链描述
- 输出：攻击向量、风险链条、严重度评估

**3. Policy Advisor Agent**

- 输入：风险分析结果 + 当前策略
- 输出：立即行动、长期措施、检测规则建议

**协作流程：**

1. Orchestrator 调度 → 决定调用哪些 Agent
2. Extractor 抽取行为链（复用已有模块）
3. RiskAnalyst 深度分析（如果调度）
4. PolicyAdvisor 生成建议（如果调度）
5. 生成协作总结文本

### 8.1 双层验证机制

**第一层：规则引擎**

**效果：** 显著降低误报率，提升命中率

### 8.2 动态权重风险评分

**自适应权重调整：**

- 正常任务（负分规则）：rule_weight=0.60，chain_weight=0.20
- 高危攻击（>50分）：rule_weight=0.35，chain_weight=0.45
- 默认：rule_weight=0.25，chain_weight=0.55

**原理：** 信任可靠的规则判断，对不确定的行为链保持谨慎

### 8.3 五层防线模型

**独创的五层防线分析框架：**

- L1 Prompt 防线 → L2 意图防线 → L3 权限防线 → L4 数据防线 → L5 执行防线
- 每层独立检测，组合评估
- 支持跨层组合攻击检测（L1+L3, L2+L4, L3+L4）

### 8.4 可信存证链

**基于国密 SM3 的哈希链：**

- 每条记录包含前一条记录的哈希值
- 形成不可篡改的审计链条
- 支持链式验证

### 8.5 多 Agent 协作

**Orchestrator + Risk Analyst + Policy Advisor**

- 总调度：根据复杂度分配任务
- 风险分析：深度分析攻击手法
- 策略建议：生成具体防御措施

## 十二、关键技术决策

### 12.1 为什么选择混合提取（LLM + Fallback）？

| 方案          | 优点     | 缺点                                       |
| ----------- | ------ | ---------------------------------------- |
| 纯 LLM       | 理解能力强  | 延迟高、成本高、可能失败 |
| 纯关键词        | 快速可靠   | 无法理解复杂语义                                 |
| **混合（本项目）** | 兼具两者优点 | 实现复杂度稍高                                  |

### 12.2 为什么使用 SQLite？

- **轻量**：无需额外安装数据库服务
- **异步支持**：aiosqlite 支持 asyncio
- **足够**：项目数据量不大，SQLite 完全满足

### 12.3 为什么使用 SM3 而不是 SHA-256？

- **国密算法**：符合国内安全合规要求
- **性能相当**：SM3 与 SHA-256 性能接近
- **fallback**：gmssl 不可用时自动回退到 SHA-256

### 12.4 为什么规则分数阈值设为 60？

- **平衡性**：既要拦截攻击，又避免误杀正常任务
- **测试验证**：经过 62 条样例 + 100 条攻击测试验证
- **可调性**：通过环境变量或配置文件可调整

## 十三、答辩常见问题准备

### Q1: 你们的系统和传统 WAF 有什么区别？

**A:** 传统 WAF 针对网络流量，我们是针对 AI Agent 的行为审计。WAF 检测 SQL 注入、XSS 等网络攻击，我们检测 Prompt 注入、数据泄露、权限提升等 Agent 特有攻击。我们的核心是**行为链分析**和**五层防线模型**。

### Q2: 误报率如何？

**A:** 通过双层验证机制（规则引擎 + 语义验证）和负分规则（正常任务排除），我们的正常任务误杀率为 0%。在 62 条审计样例中，所有正常任务都正确通过。

### Q3: LLM 提取失败了怎么办？

**A:** 我们有 Fallback 提取器作为兜底。Fallback 基于关键词匹配和正则表达式，不依赖 LLM。实际测试中，即使 LLM 超时，Fallback 也能正确提取大部分行为。

### Q4: 规则库如何更新？

**A:** 规则库存储在 YAML 文件中，支持热加载。管理员可以直接编辑 `security_rules.yaml` 文件，无需重启服务。同时支持通过 API 动态添加规则。

### Q5: 支持哪些 LLM？

**A:** 目前支持 OpenAI 兼容 API（DeepSeek、OpenAI、通义千问等）。通过配置环境变量 `LLM_BASE_URL` 和 `LLM_MODEL` 即可切换。

### Q6: 你们的创新点是什么？

**A:** 

1. **五层防线模型**：独创的 Agent 安全分析框架
2. **双层验证**：规则引擎 + 语义验证，降低误报
3. **动态权重**：根据任务类型自适应调整评分权重
4. **可信存证**：基于国密 SM3 的哈希链
5. **多 Agent 协作**：Orchestrator 调度多个专业 Agent
